# Lesson 4: 销毁语义、弱引用与生命周期陷阱

## 这节课解决什么问题

很多人理解 GC 时, 会在“判活”这一步停下。  
但工程里更容易出事故的往往是这部分:

- 对象已经不可达, 但还没彻底释放
- 弱引用什么时候失效
- 为什么对象指针会“看起来还能用, 其实已经危险”
- 哪些生命周期写法最容易埋雷

这一节就是专门把“对象死掉之后发生什么”讲清楚。

---

## 一、先把“对象死亡”拆成三个层次

这是本节最重要的基本框架。

### 1.1 逻辑层死亡

对象已经不再从根集合可达。

### 1.2 GC 状态层死亡

对象已经进入 GC 销毁 / 垃圾处理流程。

### 1.3 物理层死亡

对象的收尾和内存释放已经真正完成。

### 1.4 为什么一定要这样分

因为 UE 中大量“GC 相关怪问题”都源于把这三层混成一个瞬间。  
而真实情况是:

- 它们通常不在同一时刻发生

---

## 二、`ConditionalBeginDestroy`、`BeginDestroy`、`FinishDestroy`

### 2.1 为什么会分成多段

引擎对象销毁不是普通 `delete` 那么简单。  
一个对象可能还涉及:

- 渲染资源释放
- 其他线程协作
- 延迟清理
- 大量子状态收尾

所以 UE 必须把销毁拆成可分阶段推进的流程。

### 2.2 正确理解方式

可以把它想成:

- `ConditionalBeginDestroy`: 开始走销毁路径
- `BeginDestroy`: 做第一阶段清理
- `FinishDestroy`: 收尾到可最终释放

### 2.3 学习重点

不要去死背“哪个函数具体在什么微小时间点触发”, 而要记住:

> 对象被判死后, UE 会通过多阶段销毁流程来完成最终释放。

---

## 三、为什么弱引用是必要的

### 3.1 `TWeakObjectPtr` 的价值不在保活, 而在安全失效

弱引用不会阻止对象被回收。  
它的核心价值是:

- 当对象消失时, 引用能安全失效
- 避免你一直握着危险的裸地址

### 3.2 一个典型工程场景

你有一个 UI 面板缓存某个 Actor:

- 它不拥有这个 Actor
- 也不该保活它
- 但又希望能查询它是否还存在

这时弱引用是合适的。  
如果你用裸指针, 很容易在一次 GC 后踩内存。

---

## 四、为什么“裸指针 + 偶尔判空”经常不够

很多人会写这种代码:

```cpp
if (CachedActor)
{
    CachedActor->DoSomething();
}
```

问题在于:

- 裸指针非空, 不等于对象仍然是安全有效的 UObject

GC 之后, 裸指针可能:

- 仍然是非空
- 但已经不是合法可用对象

所以“判空”不是 UObject 生命周期安全的充分条件。

---

## 五、对象意外被回收的四大根因

### 5.1 根因 1: 引用根本没进入 GC 图

最典型:

- 裸成员指针
- native 容器里的对象指针

### 5.2 根因 2: 把弱引用误当强引用

对象没有保活, 但代码却默认它一直在。

### 5.3 根因 3: 非 UObject 持有者没有使用 `FGCObject`

对象图在 native 世界里断了。

### 5.4 根因 4: 临时对象没有被真正拥有

例如:

- 某函数里拿到了对象
- 当前帧立刻使用似乎没问题
- 但没有任何真正的持有路径

---

## 六、对象意外没被回收的四大根因

### 6.1 根因 1: 根链路还在

这是最常见的原因。

### 6.2 根因 2: `KeepFlags`

特别是编辑器态下, 某些对象默认更容易被保留。

### 6.3 根因 3: `FGCObject` 或缓存层还在报告引用

native manager 可能把对象悄悄保活了。

### 6.4 根因 4: cluster 保活

对象可能不是独自被引用, 而是跟着一批稳定对象一起活着。

---

## 七、为什么 `AddToRoot` 既有用又危险

### 7.1 它为什么有用

`AddToRoot` 可以快速把对象推入根集合。  
在这些场景下它有价值:

- 临时保活
- 调试验证“是否是 GC 导致的对象消失”
- 某些生命周期极特殊的系统对象

### 7.2 它为什么危险

因为它很容易掩盖真实问题:

- 引用图本来设计错了
- 但对象被硬塞进 root 后, 表面上“不再出 bug”
- 实际只是把错误变成了保活泄漏

### 7.3 正确态度

> `AddToRoot` 是抢救工具, 不是日常设计手段。

---

## 八、`FGCScopeGuard` 与“避免在敏感时机乱动对象图”

### 8.1 源码位置

- `Engine/Source/Runtime/CoreUObject/Public/UObject/GarbageCollection.h`

你会看到:

- `FGCScopeGuard`
- `FGCScopeTryGuard`

### 8.2 为什么这些东西存在

它们体现的是一个工程事实:

- GC 期间对象图和某些全局表处于敏感状态
- 并不是所有时机都允许你随意修改 UObject 世界

这进一步说明:

GC 不只是“后台自动清扫”, 它和运行时同步边界深度耦合。

## 九、生命周期相关的设计建议

### 9.1 区分“拥有”和“观察”

- 拥有: 用 GC 可见强引用表达
- 观察: 用弱引用表达

### 9.2 区分“短期使用”和“长期持有”

- 当前函数临时使用不等于对象被长期保活
- 想跨帧使用, 就必须有真正的持有路径

### 9.3 native 容器必须明确桥接策略

如果 native 类型持有 UObject:

- 要么别长期持有
- 要么用 `FGCObject`
- 要么重新设计到 `UObject` / 反射成员里

## 十一、常见面试题

### 面试题 1

`TWeakObjectPtr` 的核心价值是什么?

### 回答要点

- 安全观测对象是否仍然存在
- 不负责保活

### 面试题 2

为什么 `AddToRoot` 不能滥用?

### 回答要点

- 会掩盖错误引用设计
- 容易导致长期保活问题
- 只能作为特殊场景或调试工具

### 面试题 3

为什么“裸指针非空”不等于“对象安全可用”?

### 回答要点

- 因为对象可能已经进入 GC 销毁流程
- 裸指针不会自动安全失效

## 十三、本节的一个结论句

> 在 UE5 里, 真正危险的不是“GC 会回收对象”, 而是“你没有把拥有关系、观察关系和跨边界持有关系表达清楚”。
