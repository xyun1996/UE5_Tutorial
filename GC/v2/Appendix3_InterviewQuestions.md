# Appendix 3: UE5 GC 常见面试题

## 这份附录怎么用

这不是八股答案大全, 而是“常见面试问法 + 正确答题方向”。  
建议用法:

- 先自己口述回答
- 再对照这里的答题要点
- 看哪些地方你还说得模糊或不准确

这份题库刻意避免追求“标准唯一答案”, 而更强调:

- 你能不能讲对概念边界
- 你能不能说出关键机制
- 你能不能分清相近名词

---

## 一、基础概念题

### 1. UE5 GC 管理什么? 不管理什么?

#### 推荐答题要点

- GC 主要管理 `UObject` 生态中的对象生命周期
- 包括 `UObject`、`AActor`、`UActorComponent`、大量资源对象等
- 不直接管理普通 C++ `new/delete` 对象、`TSharedPtr` 对象、GPU 资源等
- UE 的内存管理不是单一系统包打天下, UObject GC 和 native 内存管理并存

#### 面试官常追问

- “那为什么 `UObject` 仍然会泄漏?”
- “为什么不是所有对象都做成 `UObject`?”

---

### 2. UE5 GC 是引用计数吗?

#### 推荐答题要点

- 不是核心依赖引用计数
- 它本质上是基于可达性分析的对象判活
- 核心问题是对象是否能从根集合沿 GC 可见引用被到达
- `FUObjectItem` 有元数据和状态, 但这不意味着生命周期判定是引用计数式

#### 加分点

- 能主动补一句 “GC 看的是引用图, 不是‘我手里还有一个裸指针’”

---

### 3. 什么是 RootSet?

#### 推荐答题要点

- 是 GC 可达性分析的起点集合
- GC 从 root 出发遍历对象图
- 只要对象能从 root 到达, 就不会在本轮被回收
- RootSet 不要和 `KeepFlags`、cluster root 混为一谈

#### 常见失分点

- 说成“RootSet 就是所有永不销毁对象”

---

## 二、引用图题

### 4. 为什么一个 `UObject*` 裸指针不能保证对象不被 GC?

#### 推荐答题要点

- 因为 GC 判断不是看“有没有地址变量存在”
- 而是看这条引用是否进入 GC 可见引用图
- 裸指针如果不在反射体系、ARO 或 `FGCObject` 报告路径里, GC 默认看不见
- 所以对象仍可能被判为不可达并回收

#### 追问方向

- “那 `UPROPERTY` 或 `TObjectPtr` 为什么重要?”

---

### 5. `TWeakObjectPtr` 和强引用的区别是什么?

#### 推荐答题要点

- `TWeakObjectPtr` 用来安全观测对象是否仍然存在
- 不会阻止对象被 GC
- 强引用则会进入 GC 引用图, 能参与保活

#### 常见失分点

- 把弱引用说成“更安全的强引用”

---

### 6. `TSoftObjectPtr` 和 `TWeakObjectPtr` 的区别是什么?

#### 推荐答题要点

- `TWeakObjectPtr` 针对当前已存在对象的弱观测
- `TSoftObjectPtr` 更偏向资源路径引用和延迟加载
- 两者默认都不是常规强保活语义

#### 加分点

- 能主动补一句 “soft 更偏资源定位, weak 更偏运行时观测”

---

### 7. `FGCObject` 解决的是什么问题?

#### 推荐答题要点

- 解决非 `UObject` 类型持有 `UObject*` 时, 如何把这些引用接入 GC 图的问题
- 通过实现 `AddReferencedObjects` 手动向 GC 上报引用
- 它是 native 世界进入 UObject GC 图的桥

#### 常见失分点

- 说成“替代 `UPROPERTY` 的通用写法”

---

## 三、主流程题

### 8. 一次 UE5 GC 周期大致会经过哪些阶段?

#### 推荐答题要点

可以用高层顺序回答:

1. 入口与预处理
2. `MarkObjectsAsUnreachable`
3. 标记 root / keep / cluster root 等初始可达对象
4. `PerformReachabilityAnalysis`
5. 收集不可达对象
6. 清理弱引用
7. `BeginDestroy` / `FinishDestroy`
8. purge / 结束回调

#### 面试提醒

- 不要把它压缩成一句“mark-sweep”
- 太短会显得你没有工程理解

---

### 9. `MarkObjectsAsUnreachable` 在做什么?

#### 推荐答题要点

- 它不是“已经确认垃圾”
- 更像先把大批对象放到“可能不可达”的候选状态
- 再由后续可达性分析把真正可达对象翻回 reachable

#### 常见失分点

- 直接说成“这一步就是找垃圾”

---

### 10. `PerformReachabilityAnalysis` 是做什么的?

#### 推荐答题要点

- 从根集合出发遍历引用图
- 沿 `ReferenceSchema`、ARO、cluster 等路径扩展
- 找出所有真正可达对象
- 这是 GC 判活的核心步骤

#### 加分点

- 能补充 schema / ARO / cluster 都会参与这一阶段

---

### 11. `BeginDestroy` 和对象真正释放是一回事吗?

#### 推荐答题要点

- 不是
- 对象判死后会进入销毁收尾阶段
- `BeginDestroy`、`FinishDestroy`、purge 可能跨多个阶段甚至多帧
- 逻辑死亡和物理内存释放是两个层次

#### 常见失分点

- 说成 “GC 一回收就直接 delete”

---

## 四、Schema 与 ARO 题

### 12. 什么是 `ReferenceSchema`?

#### 推荐答题要点

- 是类或结构体的 GC 引用布局描述
- 指导 GC 如何高效遍历对象中的强引用成员
- 包括普通引用、引用数组、结构体容器、ARO 调用点等
- 是 UE5 GC 性能和可分析性的关键基础之一

#### 面试官喜欢听到的关键词

- “预组织的遍历计划”
- “不仅靠现场慢反射”

---

### 13. `AddReferencedObjects` 在 UE5 里还有必要吗?

#### 推荐答题要点

- 有必要
- schema 解决的是大量规则化引用遍历
- ARO 解决的是反射不方便直接表达的复杂引用关系
- 两者不是替代关系, 而是协作关系

#### 常见失分点

- 说 “现在有 schema 了, ARO 基本没用了”

---

### 14. UE5 GC 为什么不只是“对象数量问题”, 还和布局有关?

#### 推荐答题要点

- 因为 traversal 需要按引用布局去访问成员
- schema 复杂度、容器层次、ARO 路径都会影响成本
- 所以性能不仅取决于对象数, 也取决于每个对象有多少引用、引用布局多复杂

#### 加分点

- 能提到 `gc.DumpSchemaStats`

---

## 五、Cluster 题

### 15. 什么是 GC Cluster?

#### 推荐答题要点

- 是对稳定对象群做的 GC 图压缩与批处理优化结构
- 它不改变对象是否该活的语义
- 主要目标是降低 traversal 成本

#### 常见失分点

- 说成 “cluster 就是另一套 GC”

---

### 16. cluster 适合什么场景?

#### 推荐答题要点

- 稳定、cooked、批量、同生命周期的对象图
- 例如一些资源对象群
- 适合子对象多、结构长期不变的场景

#### 常见失分点

- 说成 “GC 慢就都该开 cluster”

---

### 17. 为什么 cluster 会 dissolve?

#### 推荐答题要点

- cluster 依赖对象关系稳定
- 一旦内部引用或状态变得不适合继续维护 cluster
- 引擎会 dissolve 并退回普通对象处理路径

#### 加分点

- 能提到 mutable objects 或不稳定边界关系

---

## 六、Incremental 题

### 18. 什么是 incremental reachability?

#### 推荐答题要点

- 把可达性分析拆成多次迭代, 分多帧推进
- 它主要针对 traversal 主阶段
- 不是“整个 GC 全都增量化”的同义词

---

### 19. incremental reachability 和 incremental purge 有什么区别?

#### 推荐答题要点

- incremental reachability 针对前半段判活遍历
- incremental purge / begin destroy 针对后半段销毁与清理
- 它们解决的是不同阶段的尖峰问题

#### 常见失分点

- 把它们都混成一句 “增量 GC”

---

### 20. 为什么 incremental GC 需要 barrier?

#### 推荐答题要点

- 因为增量遍历跨多帧时, 对象图仍然会变化
- 新引用可能在 traversal 尚未完成时出现
- barrier 负责把新 reachable 对象及时纳入修正路径
- 否则可能错误回收本该存活的对象

#### 加分点

- 能提到 `ObjectPtr.h` 中 `UE::GC::MarkAsReachable(...)`

---

### 21. incremental GC 为什么不能保证永不卡?

#### 推荐答题要点

- 它只是把成本摊平
- 不是消灭所有成本
- pending 的 reachability / purge 可能被强制收尾
- 对象图本身过大或销毁成本过高时仍然会卡

---

## 七、调优与实战题

### 22. GC hitch 时你第一步会做什么?

#### 推荐答题要点

- 先分段定位
- 判断问题在 traversal、gather 还是 purge
- 再决定该看对象图、schema、incremental 还是 destroy

#### 常见失分点

- 一开口就说调时间预算

---

### 23. 对象“本该活却死了”时你优先查什么?

#### 推荐答题要点

- 先查引用图可见性
- 看是否用了裸指针
- 看关键引用是否进入 schema / ARO / `FGCObject`
- 看是否误用弱引用

---

### 24. 对象“本该死却没死”时你优先查什么?

#### 推荐答题要点

- 先查根链路
- 再查 `KeepFlags`
- 再查 `FGCObject`
- 再查 cluster 或缓存层保活

---

### 25. 为什么 `AddToRoot` 不能滥用?

#### 推荐答题要点

- 它会直接把对象推入根集合
- 容易掩盖错误的引用设计
- 会造成长期难发现的保活问题
- 适合作为临时保活或调试手段, 不应替代正确引用图设计

---

## 八、提高题

### 26. 为什么对象建模通常比调 GC 参数更重要?

#### 推荐答题要点

- GC 成本根源是对象数量、引用数量、布局复杂度
- 参数只能摊平或转移成本
- 结构设计能直接减少遍历规模和销毁规模

---

### 27. 如果让你解释“UE5 GC 的现代化特征”, 你会说什么?

#### 推荐答题要点

- 更强调 schema 驱动 traversal
- 有更完整的 incremental reachability 能力
- 有 cluster 等图压缩优化
- 有更明确的状态机和分阶段后处理

---

### 28. 为什么说“懂 GC 机制”和“会定位 GC 问题”不是一回事?

#### 推荐答题要点

- 懂机制是知道流程和概念
- 会定位问题则要求你能把问题先归到正确阶段
- 真正的工程能力在于把对象图、阶段、工具、配置串起来

---

## 九、答题方式建议

### 9.1 不要只给定义

例如问:

- “什么是 cluster?”

不要只说:

- “cluster 是一组对象”

更好的答法是:

- “cluster 是对稳定对象群做的 GC 图压缩与批处理优化结构, 目标是降低 traversal 成本, 不改变对象生死语义”

### 9.2 面试里多用“分层回答”

例如:

- 概念是什么
- 在流程中处于哪一段
- 最容易和什么混淆

这种答法会比单句定义更稳。
